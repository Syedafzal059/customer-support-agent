Agents should be able to retrieve Ticket details from System of Record (Assume Jira or other similar system), identify relevant information from knowledge base and address customer concerns

🚀 PHASE 2 — REDIS MEMORY + CACHE
You are a senior backend engineer.

Add a memory and caching layer that uses the same key patterns and semantics as Redis, so we can swap to a real Redis server later without rewriting call sites.

Infrastructure note (implemented default):
- Use an in-process, thread-safe hash map backend (`MemoryStore` in `redis_client.py`) — no Redis install or Docker required for local MVP.
- Data lives in RAM only; restarting the app clears memory. Optional env: `REDIS_BACKEND=memory` (default in `configs/config.yaml`).
- Future: set `redis.backend: redis` and wire a real Redis client using `redis.url` / `REDIS_URL` when you want persistence or multi-instance deployment.

Create:
- app/memory/redis_client.py  — `MemoryStore` + singleton accessor; list + string operations used by chat/cache
- app/memory/chat_memory.py   — `get_chat_history`, `append_message`, `get_cache`, `set_cache` (Redis-style keys)

Requirements:

1. Storage connection (config-driven):
   - `redis.backend: memory` (default) → use `MemoryStore`
   - Reserved: `redis.backend: redis` + URL when a real server is added

2. Chat history:
- Key: chat:{user_id}
- Store last 5 messages only
- Functions:
    - get_chat_history(user_id, store)

3. Cache:

🚀 PHASE 3 — INTENT CLASSIFIER
You are designing an LLM-based intent classification module.

Create:
- app/orchestrator/intent_classifier.py — calls LLM with history + current message
- app/llm/prompts.py — all prompts (intent few-shots live here; add RAG/ticket prompts in later phases)
- app/llm/client.py — OpenAI wrapper with structured parse (`beta.chat.completions.parse`)
- app/llm/schemas.py — Pydantic models for structured outputs (e.g. `IntentClassification`)

Requirements:

Two intents only:
- "question" — route to RAG / knowledge base (documentation, how-to, policies, general troubleshooting without a concrete ticket key).
- "ticket" — route to ticket system API (status/details of a specific issue key like ABC-123, case/ticket follow-up).

Logic:
- LLM-based (not rule-based). Provide few-shot guidance in the system prompt.
- Inputs: **recent chat history** (prior turns) + **current user message**.
- Output (structured): `intent` ("question" | "ticket"), `confidence` (0–1), `rationale` (short string for logs).

Integrate into /chat flow:
- Run classification before persisting the new user turn (history excludes current message).
- Log intent, confidence, and rationale (structured logging).

Config / secrets:
- Model name in `configs/config.yaml` under `llm.intent_classifier_model` (env override: `OPENAI_INTENT_MODEL`).
- `OPENAI_API_KEY` in `.env`; optional `OPENAI_BASE_URL` for Azure or proxies (`llm.openai_base_url`).

Keep prompts maintainable in `prompts.py`; extend the same file as new LLM features are added.

🚀 PHASE 5 — FAISS (KNOWLEDGE BASE)
You are implementing a simple RAG retrieval system.

Create:
- app/retrieval/faiss_store.py
- app/retrieval/embeddings.py
- app/retrieval/chunking.py (tiktoken-based chunk boundaries)

Requirements:

1. Load documents from:
   data/knowledge_base/

2. Chunking:
- size: 400 tokens
- overlap: 50

3. Generate embeddings via **Hugging Face** models using `sentence-transformers` (not OpenAI embeddings for this project).
   - Default **`BAAI/bge-base-en-v1.5`**: solid tradeoff between **MTEB leaderboard** retrieval quality and speed/size (~109M, 768-d). See https://huggingface.co/spaces/mteb/leaderboard and https://mteb.info/
   - Faster / lighter option: **`BAAI/bge-small-en-v1.5`** (~33M, 384-d) — set `embeddings.model_id` or env `EMBEDDING_MODEL_ID`
   - Implementation: `app/retrieval/embeddings.py` (`embed_texts`, lazy model load); config in `configs/config.yaml` under `embeddings`

4. Store in FAISS index

5. Implement:
   search(query, top_k=3)

6. Return relevant chunks as text

7. Integrate into orchestrator:
- If intent == "question" → call FAISS search

Keep it simple and working (no over-optimization).
